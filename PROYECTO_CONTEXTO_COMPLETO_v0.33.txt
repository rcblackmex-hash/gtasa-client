1. OBJETIVO DEL PROYECTO
================================================================================

Cliente multijugador para GTA San Andreas Android v2.10 (ARM64). Conecta a un
server TCP centralizado y hace que los demas jugadores aparezcan como NPCs
(peatones) en el mundo del jugador local, hijackeando peatones del pool real
del engine y sobrescribiendo su posicion cada 100ms con la pos del remoto.

Stack:
  - Mod nativo C++ ARM64 (.so) cargado por AML (Android Mod Loader v1.2.4)
  - Server TCP en Railway (yamanote.proxy.rlwy.net:19365) - protocolo simple
    de texto separado por |
  - Build via GitHub Actions (CMake + Android NDK)
  - Repo: https://github.com/rcblackmex-hash/gtasa-client

================================================================================
2. PROTOCOLO DEL SERVER
================================================================================

Cliente -> Server:
  JOIN|<name>                  al conectar
  POS|<x>|<y>|<z>              cada 1s (heartbeat con posicion del jugador)

Server -> Cliente:
  ID|<myId>                    asignado al hacer JOIN
  PLAYERS|<id>|<name>|<x>|<y>|<z>[|<id>|<name>|<x>|<y>|<z>...]
                               broadcast de remotos. CONFIRMADO en v0.33:
                               formato single (6 campos) con UN remoto.
                               El server podria también mandar batch (multi
                               remoto en un solo mensaje), nuestro parser
                               detecta ambos.
  CHAT|<msg>                   mensajes de chat
  QUIT|<id>                    notifica desconexion de un remoto
  PING                         anti-idle (lo ignoramos)

BUG/FEATURE del server observado en v0.33:
  - Server asigna g_MyId = 110 al usuario
  - PERO ademas broadcastea un PLAYERS con id=111 name="rcblackmex" (mismo
    nombre del user). Probablemente sesion zombie del mismo user. Eso es
    bug del server, no del cliente. v0.34 puede filtrar por nombre tambien
    (no solo por id) para evitar hijackearnos.

3. ARQUITECTURA DEL CODIGO (src/main.cpp)
================================================================================

3 hilos:
  - Main thread: OnModLoad/OnModUnload (entrypoint de AML)
  - NetThread: recv TCP, parsing de mensajes, envio de POS heartbeat 1Hz
  - AttackThread: resolucion de simbolos, setup, lock loop infinito 100ms

Helpers principales:
  - FindLibBase(name)              busca base de .so en /proc/self/maps
  - StripTag(addr)                 quita TBI tag (alto byte 0xb4 en android arm64)
  - IsValidPtr(addr)               rango heap razonable [0x6000000000..0x7fffffffffff]
  - IsPageMapped(addr)             mincore() check
  - SafeRead(addr, out, len)       lectura con validacion
  - InstallScanFaultHandler        handler SIGSEGV/SIGBUS con sigjmp para
                                   recovery (NECESARIO para iterar memoria)

Resolucion de simbolos via dlsym de libGTASA.so:
  _Z13FindPlayerPedi                FindPlayerPed(int) -> CPed*
  _Z17IsPedPointerValidP4CPed       IsPedPointerValid(CPed*) -> bool
  _ZN6CPools11ms_pPedPoolE          CPools::ms_pPedPool (variable global)
  _ZTV4CPed                         vtable CPed (sumar +0x10 para instances)
  _ZTV10CPlayerPed                  vtable CPlayerPed
  _ZTV8CVehicle                     vtable CVehicle
  _ZTV5CBike                        vtable CBike

4. ESTRUCTURAS Y OFFSETS CONFIRMADOS
================================================================================

CWorld players array:
  base + 0x00BDC738   ptr al CPlayerInfo del slot 0 (jugador local = CJ)
  CPlayerInfo + 0x18  ptr al CPlayerPed* (el CJ propiamente dicho)
  CPed + 0x18         ptr al CMatrix*
  CMatrix + 0x30      pos float x,y,z (12 bytes)
  CMatrix + 0x50      backref al CPlaceable (no usado en v0.30+)

CPed (Android v2.10 build):
  SIZEOF_CPED = 0x9C8 (2504 bytes) - confirmado en v0.32 derivando del log

CPool layout (confirmado v0.32, layout estandar GTA SA):
  +0x00  m_pObjects        T*       (array de objetos con TBI tag)
  +0x08  m_pFlags          uint8*   (array de flags, 1 byte por slot, bit 0x80 = free)
  +0x10  m_nSize           int      (capacidad total)
  +0x14  m_nFirstFree      int      (siguiente slot libre)
  +0x18  m_bOwnsAllocs     bool

ARM64 C++ ABI (Itanium):
  Las vtables (_ZTV*) tienen 0x10 bytes de overhead inicial:
    +0x00  offset_to_top
    +0x08  typeinfo_ptr
    +0x10  primer slot virtual <-- aqui apuntan las instancias
  Por eso scan v0.15 encontraba 0x8330F8 y dlsym da 0x8330E8: diff 0x10.

================================================================================

main.cpp tiene 757 lineas. Estructura:

```
ModInfo modinfo = {
    "com.rcblackmex.gtasaclient", "GTA SA Cliente MP",
    "rcblackmex", "0.33-remotes"
};

Constantes clave:
  kHoverZ = 5.0f           altura del demo circle
  kCircleRadius = 6.0f
  kHoverTickMs = 100       intervalo del lock loop
  kInitialWaitS = 20       espera para streaming
  kRemoteTimeoutMs = 30000 cleanup remoto stale
  kSizeofCPed = 0x9C8

struct PedSlot { uintptr_t pedPtr; uintptr_t matrixPtr; };
struct RemotePlayer {
    int id, std::string name, float pos[3], int64_t lastUpdateMs, PedSlot ped;
};

State global:
  std::map<int, RemotePlayer> g_Remotes
  std::vector<PedSlot>        g_FreePeds   <- pool de peds NO asignados
  std::mutex                  g_RemoteMutex
  int                         g_MyId

Flujo principal (AttackThreadFn):
  1. usleep 20s (streaming)
  2. FindLibBase("libGTASA.so")
  3. ResolveSymbols() via dlsym
  4. ReadLocalPlayerPos() reintentos
  5. LoadCPedPool() - double-deref con TBI strip
  6. FindPlayerPed(0) -> cjPed
  7. IteratePool() -> vector<PedSlot> filtrando CJ y validando via API
  8. g_FreePeds = peds
  9. while (g_Running) cada 100ms:
       - ReadLocalPlayerPos -> cjX, cjY, cjZ
       - TickAllPositions(cj):
           * cleanup remotos stale (>30s sin update)
           * asignar peds libres a remotos sin ped
           * para cada remoto con ped: WritePedPos(remote.pos)
           * para primeros 8 peds libres: WritePedPos(cj + circle_slot)
           * log state cada 15s

HandleMessage("PLAYERS|..."):
  - ParsePlayersMessage: itera campos de 5 en 5
  - Filtra g_MyId (no hijackearnos)
  - Crea/actualiza g_Remotes[id]
  - Si nuevo: log "[REMOTE] NUEVO id=X name=Y pos=(...)"

HandleMessage("QUIT|<id>"):
  - Libera ped del remoto, lo devuelve a g_FreePeds, borra del map

WritePedPos(p, pos):
  - IsPedPointerValid check primero
  - memcpy 12 bytes en matrix + 0x30
  - Si IsPedPointerValid retorna false -> ped murio (streaming),
    el caller lo descarta
```

8. PROBLEMAS / FRICCIONES CONOCIDAS PARA v0.34
================================================================================

A) PEDS MUEREN POR STREAMING
   Cuando un ped esta lejos del CJ (>~30-50m), el game lo libera del pool
   (streaming out). Vimos 4 peds asignados consecutivamente murieron en
   100ms cada uno hasta que uno sobrevivio. Self-healing funciona pero
   se ve glitch al user.

   Solucion v0.34 propuesta:
   - Si remoto.pos esta a >40m del CJ: NO escribir pos real, hacer hover
     "tether" del ped cerca del CJ (estilo Goliath en Half-Life)
   - O: spawn placeholder ped cerca antes de teleport
   - O: usar peds que ESTABAN cerca al iniciar (filtrar en IteratePool)

B) ECHO DEL PROPIO USUARIO (sesion zombie)
   El server broadcastea PLAYERS con id=111 name=rcblackmex pero el
   g_MyId es 110. Es otra sesion (probablemente PC + Android del mismo
   user).

   Solucion v0.34:
   - Filtrar TAMBIEN por nombre coincidente: si name == PLAYER_NAME
     y la pos esta muy cerca del CJ, ignorar como auto-echo

C) PED SE QUEDA QUIETO O CON ANIMACION DE PANICO
   Cuando reescribimos pos cada 100ms, el game cancelan animaciones AI.
   El ped puede quedar en T-pose o animacion de "caer". No hay animacion
   de "caminar" porque no estamos tocando el animator.

   Solucion v0.34/35:
   - Calcular velocity = (new_pos - old_pos) / dt
   - Si velocity > 0.5 m/s: forzar animacion de caminar
   - Si velocity > 4 m/s: forzar animacion de correr
   - Si velocity ~= 0: forzar idle
   - Requiere ResolveSymbols extra: CPed::SetAnimGroup, ePedState, etc.

D) NO HAY ROTACION
   Escribimos solo pos +0x30, no la matriz de rotacion (que esta en
   CMatrix +0x00..+0x2F = 3x3 + translation). El ped mira hacia donde
   el game decida (ultima rotacion idle).

   Solucion v0.34:
   - Calcular yaw = atan2(dy, dx) entre pos previa y actual
   - Construir matriz rotacion 3x3 alrededor de Z
   - Escribir 9 floats en matrix+0x00 (right + forward + up)

E) NO HAY NOMBRES FLOTANTES
   No se ven los nombres de los remotos. Util para multiplayer.
   Requiere hookear CFont::PrintString o usar la API de in-game text.

13. TIPS PARA EL DEBUGGING REMOTO (sin acceso fisico al device)
================================================================================

- TBI Tag de ARM64 Android: bytes altos 0xb4... son tag. Siempre
  StripTag() antes de usar como direccion real (mascara 0x00ffffffffffffff)
- mincore() para verificar si una pagina esta mapeada (evitar segfault)
- sigsetjmp/siglongjmp para recovery de segfaults durante iteracion
  de memoria
- /proc/self/maps para enumerar regiones y encontrar libGTASA.so base
- dlopen(name, RTLD_NOW | RTLD_NOLOAD) para handle de lib ya cargada
- C++ name mangling Itanium: clases empiezan con _Z, vtables _ZTV,
  variables estaticas _ZN<Class><Member>E
